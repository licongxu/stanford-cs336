# RAG 完整链路

很多人会对 RAG 产生一种错觉：只要把文档切分一下，存进向量库，再把检索结果塞进上下文，模型就做到了“基于知识库回答问题”。

这个错觉通常来自 demo 场景。

Demo 里的文档是干净的，问题是提前设计好的，答案也刚好在某个段落里。只要检索能命中那个段落，模型看起来就很聪明。这种 Demo 场景很适合向上汇报或者对外演示，甚至23年24年的时候，有些企业仅凭一些预制的问题，就能够拉到不菲的投资。

但一旦进入真实业务场景，问题会立刻变复杂：用户问法是模糊不固定的，文档质量参差不齐，知识版本互相冲突，权限需要隔离，答案需要是有来源的，成本还不能失控。

所以，做 RAG 项目时，不要去在意用什么框架，用什么 embedding 模型，关键在于**如何把一堆不稳定的文档、问题和模型输出，组织成一个可靠、可评估、可追踪的回答系统。**

### 1 常见的 RAG 系统的问题

#### 1.1 召回不到

比如用户问“报销差旅有什么限制”，文档里写的是“交通与住宿费用规范”。从语义上看是同一件事，但如果文档切分、embedding、查询改写或检索策略做得不好，系统可能根本找不到正确段落。

召回不到，是 RAG 最致命的问题之一。因为后面用于回答的模型再强，也只能基于拿到的上下文回答。**检索阶段没拿到问题相关的证据，回答阶段就只能依靠模型自身知识，甚至拿到的文本还可能污染上下文。**

#### 1.2 召回太多

为了解决召回不到的问题，很多人会把 Top-K 调大，从 5 改成 10，再改成 20，再到100。

这会带来另一个问题：召回内容变多了，但相关内容的密度下降了。模型看到一堆相似、重复、过期、甚至互相矛盾的片段，反而更容易答错。

#### 1.3 上下文污染

上下文污染指的是：系统检索到了看似相关、但实际上会误导模型的内容。

比如：

* 过期的文档被召回了；
* 草稿文档被召回了；
* 另一个部门的规则被召回了；
* 相似产品的说明被召回了；
* 用户没有权限看的内容被召回了。

这些内容一旦进入 Prompt，模型通常不会主动判断“这段材料是否应该被使用”。它会把它们当成上下文的一部分，然后生成一个看似合理但实际错误的答案。

#### 1.4 答案幻觉

很多人以为加了 RAG 就不会幻觉。其实不是。

RAG 只能降低幻觉概率，不能自动消除幻觉。如果检索结果不完整，模型可能补全缺失信息；如果上下文有冲突，模型可能选择错误的一边；如果 Prompt 没有明确要求“没有依据就说不知道”，模型可能强行给出答案。

因此，通常需要明确约束模型：**只能基于给定证据回答，证据不足时必须拒答或说明不确定。**

#### 1.5 引用不准

有些系统会在答案后面挂几个来源链接，但这些来源只是“被检索到的文档”，并不一定真的支持答案里的每一句话。这会造成一种非常危险的错觉：答案看起来有引用，但引用并不能证明答案正确。

好的引用不是“列出来源”，而是要做到：答案中的关键事实，能回溯到具体文档、具体段落，最好还能定位到页码、章节或 chunk。

它的问题也很明显：它默认“向量相似度高”就等于“能回答问题”，“Top-K 检索结果”就等于“可用上下文”，“模型看到上下文”就会“忠实回答”。

从效果上看，RAG 的核心其实是上下文工程。

所谓上下文工程，就是围绕一个目标做设计：**在模型生成答案之前，给它准备一组干净相关的证据。**

#### 1. Chunk size

Chunk 太小，语义容易断裂。比如一个条款的定义在上一段，适用条件在下一段，如果切得太碎，单个 chunk 就无法独立回答问题。

Chunk 太大，噪声又会增加。一个 chunk 里如果塞进太多主题，向量表示会被稀释，检索时可能“看起来相关”，但真正有用的信息只占很小一部分。

经验上，chunk size 不应该只按字符数决定，而应该结合文档结构。合同、论文、FAQ、API 文档、会议纪要，适合的切分方式都不一样。

好的切分不是每 512 字切一刀，而是尽量保持语义单元完整：标题、章节、条款、表格、问答对、代码块，都应该被当成结构来处理。

#### 2. Overlap

Overlap 的作用是防止关键信息刚好被切在两个 chunk 边界上。但 overlap 太大，会带来重复召回。模型看到多个高度相似的片段，不仅浪费 token，还可能把重复内容当成“更重要的证据”。

Overlap 应该服务于语义连续性，而不是粗暴增加冗余。

#### 3. Metadata

很多 demo 只存 chunk text 和 embedding。生产系统不能这么简单。

至少应该存：

* 文档 ID
* 来源系统
* 标题
* 章节路径
* 创建时间
* 更新时间
* 版本
* 权限标签
* 业务线
* 语言
* 文档类型
* 页码或段落位置

Metadata 的价值是让系统能过滤、排序和追踪。

比如用户问“最新版报销制度”，系统就应该优先召回最新版本，而不是历史版本。用户属于 A 部门，就不应该检索 B 部门的内部文档。用户问合同第 8 条，系统就应该能定位到具体条款，而不是只返回一段模糊相似的文本。

#### 4. Query Rewrite

用户问题通常不是检索友好的问题，而且往往很口语化：

* “这个能报吗？”
* “上次那个政策现在还有效吗？”
* “如果提前解约怎么办？”
* “他们之前说的那个限制是什么？”

这些问题直接拿去向量检索，效果通常不稳定。因为它们缺少主语、上下文和关键词。

Query Rewrite 的作用，是把用户问题改写成更适合检索的问题。

例如：

用户问：“这个能报吗？”

结合对话历史，可以改写成：“员工出差期间网约车费用是否可以报销？报销条件和限制是什么？”

用户问：“提前解约怎么办？”

可以改写成：“合同提前终止的条件、违约责任和通知期限是什么？”

Query Rewrite 的目的是让检索意图更清楚。

#### 5. Hybrid Search

向量检索擅长语义相似，但它不一定擅长精确匹配。

比如用户问：

* “ISO 27001 第 8.1 条是什么？”
* “订单号 AB20240517 的状态是什么？”
* “Python SDK 里 create\_session 参数怎么用？”

这些问题里有大量精确词、编号、代码、实体名。只靠向量检索，可能会错过关键词完全匹配的内容。

所以生产 RAG 常用 hybrid search，也就是把向量检索和关键词检索结合起来。向量检索负责语义召回，BM25 或倒排索引负责精确匹配，再通过合并策略或 rerank 得到最终候选结果。

#### 6. Rerank

检索通常分两阶段。第一阶段负责召回，目标是“不要漏掉可能相关的内容”。第二阶段负责重排，目标是“把真正能回答问题的内容排到前面”。

Reranker 可以是 cross-encoder，也可以是轻量模型，甚至可以用 LLM 做判断。它用来判断这个 chunk 是否真的能支持回答这个问题。

从使用效果上来判断，高质量 rerank 方法比 embedding 模型更管用。

#### 7. Context Compression

最后进入 LLM 的上下文，不应该只是 Top-K chunk 的简单拼接。

更好的做法是构造上下文，常见的做法有：

1. 去重
2. 过滤无关句子
3. 保留关键证据
4. 按主题或时间排序
5. 保留来源信息
6. 压缩长段落
7. 把冲突信息显式标出来
8. 把表格、列表、条款转换成模型更容易理解的格式

### 3 RAG 的评估

RAG 链路太长了。一个错误答案可能来自切分问题、embedding 问题、检索问题、rerank 问题、上下文构造问题，也可能来自模型生成问题。缺乏合理的评测模式，对于优化整个系统而言，你将无从下手。

Amplify Partners 的 2025 AI Engineering Report 提到，70% 的受访者正在以某种形式使用 RAG；在监控与可观测性方面，60% 的受访者使用标准 observability，超过 50% 依赖离线 eval；在评估模型和系统准确性时，人类 review 仍然是最常见方法。

RAG 评估可以拆成四类。

#### 3.1 检索评估

检索评估需要回答的问题是：**系统有没有把正确证据找出来？**

常见指标包括：

* Recall@K：正确证据是否出现在 Top-K 里
* Precision@K：Top-K 里有多少是真相关内容
* MRR：第一个正确结果排在多靠前
* nDCG：排序质量如何
* Hit Rate：是否至少命中一个正确证据

#### 3.2 生成评估

生成评估需要回答的问题是：**模型有没有基于证据正确回答？**

常见维度包括：

* 答案是否相关
* 答案是否完整
* 答案是否忠实于上下文
* 是否出现上下文之外的编造
* 引用是否支持关键结论
* 证据不足时是否会说不知道

#### 3.3 端到端评估

端到端需要评估回答的是：**用户任务有没有完成？**

比如客服场景，用户是否得到可执行解决方案；法务场景，答案是否定位到正确条款；研发文档场景，代码示例是否可运行；企业知识库场景，员工是否能根据答案完成流程。

端到端评估比单点指标更贴近业务价值，因为用户最终关心的不是 Recall@K，而是能否用这个答案解决问题。

#### 3.4 线上监控

上线后，RAG 还需要持续监控。

要记录的不只是最终答案，还包括：用户问题、改写后的 query、召回的 chunk ID、检索分数、rerank 分数、最终进入上下文的内容、模型输出、引用来源、用户反馈、token消耗等等。

当用户点踩一个答案时，回看整条链路就能知道：是没召回到？召回到了但排序低？排序对了但上下文被压缩掉？上下文正确但模型答偏了？只有链路可观测，RAG 优化才能从玄学变成工程系统。
