# 大规模向量检索

> 我们已经知道如何把文本和图片变成向量，现在面临一个工程问题——**当你有 1 亿条向量时，怎么在毫秒级找到最相似的那几条？**

假设你有一个向量数据库，存了 N 条 d 维向量。给定一个查询向量 q，你要找到与 q 最相似的 k 条结果。

**暴力搜索**（Brute-force / Flat Search）的时间复杂度：**O(N × d)。**

**当面临超大规模的向量时和高并发时，系统会有无法忍受的响应时间。**

**所以，我们需要一种方法，不遍历所有向量，就能找到"大概率是最相似的"那几条。**

### 1 近似最近邻（ANN）

**ANN**（Approximate Nearest Neighbor）的核心是一个交易：

> **用一点点精度，换巨大的速度提升。**

什么叫"一点点精度"？假设暴力搜索返回的 top-10 是完美答案，ANN 返回的 top-10 可能有 9 个和完美答案重合——这就是 **90% 的召回率**（Recall@10）。

ANN 算法的分类大致有三条路线：

* 基于划分： 把空间切成小块，只搜相关的块，如IVF（Faiss）、KD-Tree
* 基于树：用树结构递归划分空间，如Annoy（随机投影树）
* 基于图：构建临近图，沿图搜索，如HNSW、NSG、DiskANN
* 基于哈希：把向量映射成短编码，如LSH

### 2 Faiss

**Faiss**（Facebook AI Similarity Search）是 Meta 开源的向量检索库，几乎是工业界的事实标准。

它不是一个单一算法，而是一个**索引工具箱**，你可以像搭积木一样组合不同的策略。

#### 2.1 IndexFlat

IndexFlat 是最简单的暴力搜索基线

```python
import faiss
import numpy as np

d = 768          # 向量维度
N = 100000       # 向量数量

# 生成随机数据
vectors = np.random.random((N, d)).astype('float32')

# 创建暴力搜索索引
index = faiss.IndexFlatL2(d)   # L2 距离
# 或者 faiss.IndexFlatIP(d)   # 内积（余弦相似度用这个，但要先归一化）

index.add(vectors)             # 添加向量

# 搜索
query = np.random.random((1, d)).astype('float32')
distances, indices = index.search(query, k=10)  # 返回 top-10
```

* **IndexFlatL2**：用欧氏距离（L2）暴力搜索
* **IndexFlatIP**：用内积（Inner Product）暴力搜索

#### 2.2 IVF：倒排文件索引

**IVF**（Inverted File Index）的核心思想是：**把 N 个向量先用 K-Means 分成 nlist 个簇，查询时只搜最近的几个簇，而不是全部。**

#### 2.3 PQ：乘积量化

IVF 解决了速度问题，但内存问题还在。1 亿条 768 维 float32 向量还是要 286 GB。

**PQ**（Product Quantization，乘积量化）的目标是：**把每个向量从几百字节压缩到几十字节，同时尽量保留距离信息。**&#x5B83;不是一次性量化整个高维向量（这通常会导致分辨率不佳），而是首先将向量分解成多个低维子向量。

假设有一个 $$D$$ 维空间中的原始向量 $$\mathbf{x}$$（$$\mathbf{x} \in \mathbb{R}^D$$）。PQ 将这个向量分成 $$M$$ 个不同的子向量，每个子向量的维度为 $$D/M$$。为简单起见，我们假设 $$D$$ 可以被 $$M$$ 整除。

$$
\mathbf{x} = \left[ \mathbf{x}_1 \mid \mathbf{x}_2 \mid \cdots \mid \mathbf{x}_M \right]
$$

这里，每个 $$\mathbf{x}_m$$（$$1 \le m \le M$$）都是一个维度为 $$D/M$$ 的子向量。比如，一个 768 维的向量可以分成 $$M=48$$ 个子向量，每个维度为 $$D/M=16$$。

核心思想是独立地量化这 $$M$$ 个子空间中的每一个。对于每个子空间 $$m$$，我们创建一个独立的码本 $$C^m$$。这个码本包含 $$k^*$$ 个代表向量（通常称为聚类中心），它们是该子空间特有的。这些聚类中心通常是使用 k-means 聚类算法，从与该子空间对应的子向量训练集中学习得到的。

$$
C^m = \left\{ \mathbf{c}_{m,1}, \mathbf{c}_{m,2}, \ldots, \mathbf{c}_{m,k^*} \right\}
$$

其中 $$\mathbf{c}_{m,j} \in \mathbb{R}^{D/M}$$。

生成 $$M$$ 个码本后，任何原始向量 $$\mathbf{x}$$ 都可以被编码。向量 $$\mathbf{x}$$ 的每个子向量 $$\mathbf{x}_m$$ 被映射到其在对应码本 $$C^m$$ 中最近聚类中心的索引（ID）。令 $$q_m(\mathbf{x}_m)$$ 为查找子向量 $$\mathbf{x}_m$$ 在 $$C^m$$ 中最近聚类中心索引的函数：

$$
q_m(\mathbf{x}_m) = \underset{j \in \{1,\ldots,k^*\}}{\arg\min} \left\| \mathbf{x}_m - \mathbf{c}_{m,j} \right\|^2
$$

原始向量 $$\mathbf{x}$$ 随后由 $$M$$ 个聚类中心索引序列表示：

$$
\text{编码}(\mathbf{x}) = \left[ q_1(\mathbf{x}_1), q_2(\mathbf{x}_2), \ldots, q_M(\mathbf{x}_M) \right]
$$

每个索引 $$q_m(\mathbf{x}_m)$$ 需要 $$\log_2(k^)$$ _位。如果我们选择 \$$k^_=256\$$，每个索引需要 8 位（1 字节）。因此，整个 D 维向量 $$\mathbf{x}$$ 被压缩成仅仅 M 字节。与存储原始 D 维 float32 向量（需要 $$D \times 4$$ 字节）相比，这代表了很大的压缩比，通常根据 $$D$$ 和 $$M$$ 的不同，范围在 10 倍到 100 倍之间。对于我们 768 维的例子，当 $$M=48$$ 时，压缩后的大小是 48 字节，而原始大小为 $$768 \times 4 = 3072$$ 字节，压缩因子为 64 倍。

### 3 Annoy：随机投影树

**Annoy**（Approximate Nearest Neighbors Oh Yeah）是 Spotify 开源的 ANN 库。

Annoy的设计目标非常明确：

* **低内存占用**：索引可以通过内存映射（mmap）方式加载，多个进程可以共享同一份索引，极大降低内存开销
* **高查询速度**：通过树结构快速缩小搜索范围
* **易于使用**：简单的API，便于集成到生产系统

#### 3.1 单棵随机投影树的构造

Annoy 的核心数据结构是**随机投影树**（Random Projection Tree），构造过程如下：

```
所有向量（叶子节点还没分裂前）
    │
    ▼  随机选两个点 A 和 B
    │
    ▼  计算 A、B 的中垂面（超平面），把空间一分为二
    │
    ├── 左子树：在超平面"A 侧"的向量
    └── 右子树：在超平面"B 侧"的向量
    │
    ▼  对左、右子树递归执行上述过程
    │
    ▼  直到每个叶子节点包含的向量数 ≤ K（比如 K=100）
```

图示：

```
        ┌─────────────────────┐
        │      全部向量         │
        │    ·  ·    ·  ·     │
        │  ·    ·  ·    ·  ·  │
        │    ·   /   ·  ·     │   ← 随机超平面把空间切开
        │  ·   /  ·    ·      │
        │    /  ·    ·   ·    │
        └──/──────────────────┘
          /
    左半空间          右半空间
    ┌──────┐        ┌──────┐
    │· · · │        │· · · │
    │ · ·  │        │ · · ·│
    └──────┘        └──────┘
       │                │
       ▼                ▼
    继续切...        继续切...
```

查询时，从根节点开始，判断查询向量在超平面的哪一侧，一路走到叶子节点，返回叶子里的向量作为候选。

#### 3.2 问题：边界上的点怎么办？

单棵树有一个明显的缺陷：如果查询点恰好在分割超平面附近，最近邻可能被分到了另一侧。

```
        ─────────/──────────
              · /  · ← 最近邻
              /
          q · /          ← 查询点在左边，但最近邻在右边！
              /
        ─────────/──────────
```

#### 3.3 解决方案：多棵树的集成

Annoy 的解决方法简单粗暴——**建多棵树，每棵树用不同的随机超平面**：

```
Tree 1: 用超平面 H1 切分 → 查询落入叶子 L1 → 候选集 C1
Tree 2: 用超平面 H2 切分 → 查询落入叶子 L2 → 候选集 C2
Tree 3: 用超平面 H3 切分 → 查询落入叶子 L3 → 候选集 C3
...
Tree T: 用超平面 HT 切分 → 查询落入叶子 LT → 候选集 CT

最终候选集 = C1 ∪ C2 ∪ C3 ∪ ... ∪ CT（去重）
在候选集上暴力搜索 → 返回 top-k
```

因为每棵树的随机超平面不同，同一个点在不同树中被分到不同的叶子。多棵树的并集大大降低了"漏掉最近邻"的概率。

**树的数量**是精度与速度的旋钮：

* 树越多 → 候选集越大 → 召回率越高 → 但搜索越慢、内存越多
* 通常 10\~100 棵树就够用

#### 3.4 使用示例

```python
from annoy import AnnoyIndex
import numpy as np

d = 768
N = 1000000

# 构建索引
index = AnnoyIndex(d, 'angular')  # 'angular' = 余弦距离, 'euclidean' = L2

for i in range(N):
    v = np.random.random(d).astype('float32')
    index.add_item(i, v)

index.build(n_trees=50)       # 构建 50 棵树
index.save('index.ann')       # 保存到磁盘

# 加载与查询
index = AnnoyIndex(d, 'angular')
index.load('index.ann')       # mmap 加载，几乎不占内存

results = index.get_nns_by_vector(query, n=10, search_k=5000)
# search_k：总共检查多少个节点（越大越精确）
```

#### 3.5 Annoy 的优势与局限

**优势：**

* **只读索引 + mmap 加载**：索引文件可以在多个进程间共享，加载几乎不额外占内存
* **极其轻量**：依赖少，编译简单，适合嵌入式或资源受限的环境
* **可预测的内存占用**：内存 ≈ 原始向量大小 × 树的数量
* **适合静态数据**：一次构建，多次查询

**局限：**

* **不支持动态增删**：索引构建后不能添加新向量（需要重建）
* **精度不如 HNSW**：在同等内存下，召回率一般低于 HNSW
* **高维向量效果下降**：随机投影在高维空间的分割效率递减

### 4 HNSW：多层导航小世界图

**HNSW**（Hierarchical Navigable Small World）是目前综合性能最强的 ANN 算法之一。它基于图结构，核心灵感来自"小世界网络"。

#### 4.1 小世界网络的直觉："六度分隔"

1960 年代，社会心理学家 Stanley Milgram 做了一个著名实验：让美国中西部的人通过"认识的人"把一封信传递给波士顿的一个陌生人。结果发现，平均只需要 **6 次传递**就能到达——这就是"六度分隔"理论。

社交网络就是一个**小世界网络**：

* 大多数人只认识附近的人（**局部连接**）
* 但总有少数人认识远方的人（**长程连接**）
* 这两种连接的结合，让任意两人之间的距离很短

**HNSW 的核心直觉就是：把向量之间的邻近关系构建成一个类似社交网络的图，然后在图上"传话"一样地搜索。**

#### 4.2 NSW：可导航小世界图

在讲"H"（层次化）之前，先理解基础版 **NSW**（Navigable Small World）。

**图的构建**

每个向量是图中的一个节点。插入新节点 v 时：

1. 从图中已有的某个入口点出发
2. 贪心地沿着图走，不断跳到离 v 更近的邻居
3. 到达局部最优后，把 v 与它附近的 M 个节点连边

```
已有图中的节点：A—B—C—D—E

插入新节点 v：
1. 从入口 A 出发
2. A 的邻居里，B 离 v 更近 → 跳到 B
3. B 的邻居里，C 离 v 更近 → 跳到 C
4. C 的邻居里没有比 C 更近的了 → 停下
5. 把 v 与 C、B、D 连边（M=3）

结果：A—B—C—D—E
            |\ |
            v──┘
```

**查询过程**

和构建一样的贪心搜索：

1. 从入口点出发
2. 检查当前节点的所有邻居
3. 跳到离查询向量最近的邻居
4. 重复，直到无法再前进（局部最优）
5. 维护一个候选列表，返回 top-k

#### 4.3 从 NSW 到 HNSW：加上层次结构

纯 NSW 有一个问题：图中所有节点都在同一层，早期插入的节点因为连接多会变成"高速公路"，但随着图增大，搜索可能需要经过很多跳才能到达目标区域。

**HNSW 的解决方案：构建多层图，高层做粗定位，低层做精搜索。**

```
Layer 3 (最稀疏):    [A] ─────────────── [G]
                      │                    │
Layer 2:         [A] ─── [D] ──── [G] ─── [J]
                  │       │        │       │
Layer 1:     [A]─[B]─[D]─[E]─[G]─[H]─[J]─[K]
                  │   │   │   │   │   │   │
Layer 0 (全部):  所有 N 个节点都在这一层，连接最密集
```

**层的分配规则**

每个节点被分配到哪一层由随机过程决定：

* **每个节点都在 Layer 0**
* 以概率 p（通常 p = 1/ln(M)）升到 Layer 1
* 以概率 p² 升到 Layer 2
* 以此类推...

这产生了一个**指数递减**的层次结构：

* Layer 0：N 个节点（全部）
* Layer 1：约 N/M 个节点
* Layer 2：约 N/M² 个节点
* ...

越高层节点越少，连接跨度越大——就像地图的不同缩放级别。

**查询流程**

```
查询向量 q

Layer 3: 从入口节点开始，贪心搜索 → 找到该层最近的节点 P3
            │
            ▼  下沉到 Layer 2，从 P3 开始
Layer 2: 贪心搜索 → 找到更精确的节点 P2
            │
            ▼  下沉到 Layer 1
Layer 1: 贪心搜索 → 找到 P1
            │
            ▼  下沉到 Layer 0
Layer 0: 在最密集的图上贪心搜索 → 找到 top-k 近邻
```

**类比**：就像你在地图上找一个餐馆——

* 先看世界地图（Layer 3）：确定在亚洲
* 再看国家地图（Layer 2）：确定在中国上海
* 再看城市地图（Layer 1）：确定在浦东新区
* 最后看街道地图（Layer 0）：找到具体地址

每一层都在缩小搜索范围，而不是在最细粒度的地图上从头找起。

HNSW 之所以能接近暴力搜索的召回率，有几个关键原因：

**1. 图结构天然保留了邻近关系**

和 IVF 的硬划分不同，图中的边直接连接了相似的向量。不存在"被分到错误的簇"的问题。

**2. 贪心搜索 + beam search**

实际实现中，HNSW 不只维护一个当前最优节点，而是维护一个大小为 ef 的候选列表（类似 beam search）。ef 越大，搜索越彻底。

```
ef = 1   → 纯贪心，最快但可能陷入局部最优
ef = 50  → 同时追踪 50 个候选路径，大幅提升召回率
ef = 500 → 接近暴力搜索的精度
```

**3. 层次结构避免了局部最优**

高层的长程连接让搜索可以快速跳到目标区域，避免在低层的局部区域里打转。

但是 HNSW 的高召回率不是免费的：

**内存占用大**：每个节点需要存储它的邻居列表。假设每个节点平均有 M 个邻居，每个邻居 ID 用 4 字节：

```
额外内存 = N × M × 4 bytes × 层数
1 亿节点，M=32 → 约 12.8 GB 仅用于存边（还没算向量本身）
```

加上原始向量（286 GB），总内存可达 **\~300 GB**。不做压缩的话，HNSW 是最吃内存的。

**构建索引慢**：每插入一个节点都要做一次搜索来找邻居，构建 1 亿条向量的索引可能要**数小时**。

**不支持高效的向量压缩**：原生 HNSW 存的是完整向量。虽然可以和 PQ 结合（如 Faiss 的 IndexHNSWPQ），但效果不如 IVF+PQ 那么自然。
